## Модификация алгоритма

### Неожиданные трудности

Не смотря на то, что алгоритм для LU разложения имеет 3 вложенных цикла, внешние 2 распараллеливать нормально не получиться. Проблема заключается в том, что разложение должно идти <ins>**строго по порядку**</ins>.
Если порядок будет нарушен, то результат вычисления окажется неверным. Директива `order` также не поможет, т.к. главная часть цикла требует порядка, как следствие - это лишь сильно замедлит алгоритм из-за ждущих друг друга потоков.

Не смотря на всё это есть части, где директивы `omp` будут полезны:
Для получения следующего элемента матриц U и L используется сумма, цикл которой отлично подходит для редукции

```cpp
#pragma omp parallel for shared(u, l, m) reduction(+:sum)
for (long k = 0; k < i; k++) {
    sum += l(i, k) * u(k, j);
}
```

Подобное справедливо и для вычисления самого определителя через произведения элементов, стоящих на диагоналях матриц L и U

```cpp
#pragma omp parallel for shared(u, l) reduction(*:det)
for (long i = 0; i < l.rows(); i++) { 
    det *= (l(i, i) * u(i, i));
}
```

### Наивный алгоритм

Этот алгоритм и так не отличается производительностью. И параллельность тут делает только хуже ввиду рекурсивной природы разложения по строке/столбцу.
